## 1.4 容错降级链路

```
正常路径： 请求 → Service → Repository → DB/API → 响应 ✓

降级路径（按优先级）：
  1. DB 故障  → 内存 LRU 缓存 → 本地 JSON 兜底
  2. AI 超时  → 降级为静态路书 + 友好提示
  3. 天气 API 限流 → 内存缓存旧数据
  4. LLM 费用超限 → 关停 AI 模块，基础功能正常
  5. 进程异常 → Crontab 监控自动重启 + 邮件告警
```

## 2.2 后端分层调用规范

```
Router（路由层）
   │  职责：请求参数校验、响应序列化、HTTP 异常处理
   │  禁止：直接访问数据库、调用第三方 API、包含业务逻辑
   ▼
Middleware（中间件层）
   │  职责：JWT 验证、游客识别、IP/用户限流
   │  执行时机：Router 之前自动执行
   ▼
Service（业务层）
   │  职责：业务编排、权限校验、降级策略、数据聚合
   │  依赖：仅依赖 Repository 层
   │  禁止：直连数据库、直连第三方 API
   ▼
Repository（数据层）
   │  职责：统一数据访问接口，屏蔽 DB/API/Cache 差异
   │  实现：DB 优先 → LRU 缓存 → JSON 兜底 三级策略
   ▼
AI Module（AI 模块）
   │  职责：LLM 调用、ToolCall 编排、输出校验、上下文管理
   │  调用方式：由 Service 层通过 asyncio 异步触发，不阻塞主线程
   │  数据来源：仅通过 Service → Repository 获取，不直连任何数据源
```

```

│  │ LLM 客户端│  │工具注册表│  │输出校验器│  │ 成本守卫     │ │
│  │(DeepSeek)│  │(3 个工具)│  │(双层校验)│  │(预扣+限流)   │ │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  └──────┬──────┘ │
│       │             │             │               │         │

│     └────────────┘ └──────────────┘                         │

## 3.2 Agent 核心执行流程（行程生成）

```
Step 1 ─ 接收请求
  │  Router → ai_service.generate_plan()
  │  参数：{ city, days, style, budget, companions }
  │
  ▼
Step 2 ─ 数据准备（Repository 层拉取，非 AI 直连）
  │  ├─ city_repo.get_plan(city)        → 标准路书
  │  ├─ weather_repo.get_current(city)  → 实时天气
  │  └─ poi_tool.check_status(spots)    → 景点实时状态
  │
  ▼
Step 3 ─ 异步 LLM 调用（asyncio，不阻塞主线程）
  │  ├─ 组装 System Prompt（模板 + 实时数据）
  │  ├─ 注册可用 Tool：get_weather / check_poi / read_plan_kb
  │  ├─ 发送请求到 DeepSeek API（max_tokens=1024）
  │  │
  │  ├─ ToolCall 循环（最多 3 轮）：
  │  │   ├─ LLM 返回 function_call → 执行工具 → 结果回传
  │  │   └─ 3 轮后强制结束，返回当前结果
  │  │
  │  └─ 超时 30s 自动降级
  │
  ▼
Step 4 ─ 双层输出校验
  │  ├─ 第一层：JSON 格式校验
  │  │   ├─ json.loads() 合法性检查
  │  │   ├─ 自动修复：补逗号、补括号、截断修复
  │  │   └─ 失败 → 降级静态路书
  │  │
  │  └─ 第二层：内容合理性校验
  │      ├─ 必填字段检查（day/spot/activity）
  │      ├─ 景点时间顺序合理性
  │      ├─ 预算范围检查
  │      ├─ 违禁词 + 虚假内容筛查
  │      └─ 失败字段 → 剔除 + 标记
  │
  ▼
Step 5 ─ 缓存 + 返回
  │  ├─ 相同参数 → 写入 LRU 缓存（1h TTL）
  │  ├─ 记录 ai_usage 统计（用户/IP、Token、费用）
  │  └─ 返回标准化 JSON 给前端
  │
  ▼
Step 6 ─ 异常兜底链
     ├─ AI 超时/失败 → 降级为静态路书 + 提示「AI 暂时不可用，展示标准行程」
     ├─ 校验不通过 → 部分展示 + 标记不可信条目
     └─ 费用超限   → 直接拒绝 + 提示「AI 服务维护中，明日恢复」
```

## 5.6 接口版本管理策略

```
┌────────────────────────────────────────────────────────────┐
│                    接口版本管理                              │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  REST API 版本：                                            │
│    采用 URL 前缀 /api/v1/...，大版本变更时升级 v2            │
│    当前项目规模小，暂用 /api/ 即可，后续平滑迁移             │
│                                                            │
│  Python 内部接口：                                          │
│    使用 Pydantic BaseModel 约束输入输出                      │
│    通过单元测试锁定契约，CI 中执行 mypy 类型检查              │
│                                                            │
│  TypeScript 前端类型：                                       │
│    与后端 Pydantic Schema 一一对应                           │
│    类型文件统一放在 frontend/src/types/api.ts               │
│    后端接口变更 → 同步更新该文件                             │
│                                                            │
│  契约测试（Contract Test）：                                │
│    关键接口（行程生成、行程同步）编写契约测试                │
│    后端改返回格式 → 测试失败 → 强制前端同步更新              │

└────────────────────────────────────────────────────────────┘
